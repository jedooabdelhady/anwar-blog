1. نظرة عامة على المشروع

علم تأويل الرؤى (sahaarr299.com) هو موقع/مدوّنة عربية تتعامل مع تأويل الأحلام والرؤى ضمن طابع فنّي ثمودي/نجدي. الموقع يدمج بين:

  • صفحات تعريفية ومحتوى مدوّنة (مقالات، تصنيفات).
  • بوّابات تفاعلية لإرسال رسائل واستفسارات من الزوّار.
  • لوحة تحكم Sanity Studio مدمجة في نفس الدومين /studio.
  • نظام رد آلي من Studio يُرسل البريد الإلكتروني للعميل عبر Resend.
  • صفحة محادثة خاصّة بكل عميل عبر رابط فريد /inquiry/{token}.

5. لوحة التحكم Sanity Studio

تُتاح اللوحة على /studio داخل نفس الدومين. يمكن للإدارة:

  • إنشاء/تعديل المقالات، التصنيفات، الكتّاب، الإعدادات.
  • مراجعة الرسائل المُستلَمة من نماذج الموقع.
  • كتابة الردّ على الرسائل ونشره — يُرسَل تلقائيًا للعميل بالبريد.

أنواع المستندات (Schema Types)

post — المقال

  • title, slug, excerpt, body (Portable Text)
  • image + alt, author (مرجعي)، category (مرجعي)
  • publishedAt, seo

category — التصنيف

  • title, slug, description
  • يُستخدم لتصفية المكتبة /blog?category=...

author — الكاتب

  • name, avatar, bio

siteSettings — إعدادات الموقع

  • نصوص الصفحات المعروضة في الـ Hero والـ Footer

submission — الرسالة المُستقبَلة

الحقلالنوعالوصف

namestringاسم العميل
emailemailبريد العميل (اختياري)

6. تدفّق الرسائل والردود (Email Flow)

1) العميل يُرسل رسالة من الموقع

  1. يملأ العميل الفورم في /forms/inquiry (أو public-vision/private-vision).
  2. الفورم يُرسل POST إلى /api/submissions بصيغة JSON.
  3. الـ API يُولِّد accessToken فريد (UUID v4).
  4. يحفظ الـ submission في Sanity مع accessToken و status: "new".
  5. يُرسل إشعارًا لك (المسؤول) عبر Resend على CONTACT_EMAIL_TO.
  6. يعيد للفورم { ok: true, id, accessToken }.
  7. الفورم يعرض رسالة نجاح + زر "فتح صفحة المحادثة" → /inquiry/{accessToken}.

2) أنت تردّ من Studio

  1. تفتح Studio على /studio → "الرسائل الواردة".
  2. تفتح الرسالة الجديدة → تكتب الردّ في حقل "ردّ المسؤول".
  3. (اختياري) تغيّر الحالة لـ "تم الرد".
  4. تضغط Publish.

3) Sanity Webhook يطلق تلقائيًا

  1. Sanity يكتشف التغيير ويستدعي webhook المُسجَّل على /api/sanity-webhook.
  2. الـ API يتحقق من توقيع HMAC-SHA256 باستخدام SANITY_WEBHOOK_SECRET.
  3. يقرأ الـ submission كاملًا (يجلب accessToken من Sanity مباشرة).
  4. إذا كان replyMessage موجود و replySentAt فارغ:
    • يُرسل البريد للعميل عبر Resend (HTML مع تنسيق RTL + زر للموقع).
    • BCC نسخة لك على CONTACT_EMAIL_TO.
    • يضع replySentAt = الآن، status = "answered".
  5. إذا فشل Resend (مثل خطأ التحقق) → يكتب الخطأ في replyError.

4) العميل يستلم الرد

  • يصله إيميل من no-reply@sahaarr299.com فيه نصّ الرد + زر "عرض المحادثة على الموقع".
  • الزر يفتح صفحة /inquiry/{accessToken} ويرى الرسالة الأصلية + الرد.
  • يمكنه العودة للرابط في أي وقت (الصفحة بـ robots: noindex فلا تظهر في Google).
الـ Idempotency: الـ webhook لا يُعيد إرسال الإيميل لو replySentAt متعبّى، حتى لو أعدت Publish.

8. إعداد Resend + Namecheap DNS

الخطوات اللي تمّت مرة واحدة عند إنشاء الـ domain:

الخطوة 1: شراء الـ domain

تم شراء sahaarr299.com من Namecheap.

الخطوة 2: إضافة الـ domain في Resend

  1. الدخول إلى resend.com/domains → Add Domain.
  2. كتابة sahaarr299.com واختيار region: Ireland (eu-west-1).
  3. Resend يعرض 3-4 records مطلوب إضافتها.

الخطوة 3: ضبط Namecheap DNS

في Namecheap → Domain List → Manage → Advanced DNS:

  1. تغيير MAIL SETTINGS من "Email Forwarding" إلى "Custom MX" (لكشف خيار MX).
  2. إضافة الـ records التالية:
TypeHostValuePriority
TXT (SPF)sendv=spf1 include:amazonses.com ~all—
TXT (DKIM)resend._domainkeyالمفتاح الطويل من Resend (p=MIGfMA...)—
MXsendfeedback-smtp.eu-west-1.amazonses.com10
TXT (DMARC)_dmarcv=DMARC1; p=none;—
قاعدة Namecheap المهمة: في خانة Host اكتب الـ subdomain فقط بدون الـ domain. مثلًا send.sahaarr299.com تكتب في الخانة send فقط — Namecheap يُكمل تلقائيًا.

الخطوة 4: تأكيد التحقّق

في Resend → الـ domain → اضغط "I've already added the records". الانتظار 5-30 دقيقة حتى تظهر جميع الـ records خضراء ✅.

9. إعداد Vercel ونشر الموقع

ربط GitHub

الـ project مربوط على فرع main من jedooabdelhady/anwar-blog. كل push على main يُطلق deployment تلقائي.

ربط الـ Custom Domain

  1. Vercel → Project → Settings → Domains → Add → sahaarr299.com.
  2. Vercel يطلب إضافة A record:
    • Type: A، Name: @، Value: 216.198.79.1
  3. وCNAME للـ www: Name: www، Value: cname.vercel-dns.com.
  4. إضافتها في Namecheap → Advanced DNS بنفس الأسلوب السابق.
  5. حذف CNAME www → parkingpage وURL Redirect @ → http://www.sahaarr299.com من Namecheap (records افتراضية).
  6. الانتظار حتى Vercel يعرض Valid Configuration ويصدر SSL تلقائيًا.
  7. تحديد sahaarr299.com كـ Primary Domain.

متغيّرات البيئة في Vercel

أضفها من Settings → Environment Variables (راجع قسم 7).

إعادة النشر بعد تعديل env

أي تعديل على متغيّر بيئة يستلزم Redeploy:

  1. Deployments → آخر deployment أخضر → ⋯ → Redeploy.
  2. إلغاء علامة "Use existing Build Cache" لضمان قراءة المتغيّرات الجديدة.

11. سجل التعديلات الكامل (Commits)

التعديلات مرتّبة من الأحدث للأقدم. كل commit مستقل ويمكن مراجعته على GitHub.

sanity-webhook: stop overwriting fresh accessTokens 5ee8204

إصلاح bug خطير: الـ backfill كان يولّد accessToken جديد ويـoverwrite الموجود لو الـ webhook projection ناقصة. النتيجة: روابط العملاء تنكسر بعد كل publish. الحل: قراءة الـ token من Sanity مباشرة قبل اللجوء لـ backfill.

inquiry/[token]: show reply as soon as admin saves it 44a0a05

الصفحة كانت تشترط replySentAt لعرض الرد. لو فشل إرسال الإيميل، العميل يبقى على "لم يَرِد الرد بعد" للأبد. الآن: تُعرض كتابة الرد فور حفظها، ووقت الإرسال يظهر فقط إذا كان موجودًا.

fix(inquiry): bind writeClient.fetch and skip CDN f702dda

إصلاحان لـ 404 في صفحة المحادثة: (1) تقييد this عند cast لـ writeClient.fetch (كان يرمي Cannot read properties of undefined '#httpRequest'). (2) استخدام writeClient بدل client لتجاوز تأخير CDN للقراءة فور الكتابة.

fix(inquiry): TS overload for client.fetch on Sanity v7 adb054c

حلّ خطأ TypeScript في build Vercel: Sanity v7 يفسّر template literals كـ QueryWithoutParams ويرفض ad-hoc params. الحل: إخراج الـ query لـ queries.ts وcast على fetch مع params arbitrary.

inquiry/[token]: customer-facing conversation page 26aa8d1

إضافة الميزة الكبرى: صفحة محادثة خاصة بكل عميل عبر رابط فريد (UUID). تتضمّن: حقل accessToken في الـ schema، توليده في /api/submissions، عرضه في ContactForm كزر، صفحة /inquiry/[token] بـ noindex، backfill في webhook، وزر CTA في إيميل الرد.

resend: surface response errors instead of silently succeeding a2bbdeb

Resend SDK يُرجع { data, error } لـ failures بدلًا من رميها كـ exception. الكود القديم كان يـ await دون فحص error، فيظن النجاح. الآن: فحص result.error وكتابته في replyError ليظهر في Studio.

reply-from-studio: send reply email when admin saves replyMessage 5d1f62e

الميزة الأساسية للرد من Studio: حقول جديدة (replyMessage، replySentAt، replyError) في schema، endpoint /api/sanity-webhook مع التحقق من توقيع HMAC، إرسال البريد عبر Resend، تحديث الـ doc.

bg: pin wallpaper to viewport, remove drift animation 094b7e0

الخلفية مثبّتة على الـ viewport ولا تتحرك مع scroll. تم نقلها لـ body::before مع position: fixed لتفادي مشكلة background-attachment: fixed في iOS Safari. إزالة الـ drift animation.

12. كيف تختبر النظام

الاختبار اليدوي الكامل (E2E)

  1. افتح https://sahaarr299.com/forms/inquiry من جهاز عميل (مش حسابك).
  2. املأ الفورم بإيميل ثانوي عندك (مش إيميل حساب Resend).
  3. اضغط "أرسل استفسارك".
  4. تظهر صفحة نجاح فيها زر "فتح صفحة المحادثة".
  5. اضغط الزر — يفتح /inquiry/{token} وتشوف رسالتك + "لم يَرِد الرد بعد".
  6. افتح /studio → الرسائل الواردة → الرسالة الجديدة.
  7. اكتب رد في "ردّ المسؤول" → غيّر الحالة لـ "تم الرد" → Publish.
  8. خلال ثوانٍ:
    • ✅ الإيميل يصل للعميل من no-reply@sahaarr299.com
    • ✅ تصلك نسخة BCC على CONTACT_EMAIL_TO
    • ✅ في Studio: حقل "أُرسل الرد في" يتعبّى تلقائيًا
    • ✅ في صفحة المحادثة (refresh): الرد يظهر
    • ✅ في إيميل العميل: زر "عرض المحادثة على الموقع" يُحيل لنفس الصفحة

اختبار الـ build محليًا

cd anwar-blog
npm install
npm run build

لازم يخرج ✓ Compiled successfully بدون أخطاء TypeScript.

تشغيل الـ dev server محليًا

npm run dev

الموقع يفتح على http://localhost:3000.

13. استكشاف الأخطاء

الإيميل لم يصل للعميل

  1. افتح Vercel → Logs → فلتر /api/sanity-webhook.
  2. لو الـ status = 500 مع رسالة Resend send failed: validation_error: You can only send testing emails…:
    • قيمة CONTACT_EMAIL_FROM لسه onboarding@resend.dev. حدّثها لـ no-reply@sahaarr299.com وredeploy.
  3. لو الـ status = 401:
    • SANITY_WEBHOOK_SECRET في Vercel مختلف عن الـ Secret في Sanity webhook. ضبّطهم.
  4. لو الـ status = 200 لكن مفيش إيميل:
    • اطلب من العميل فحص الـ Spam folder.
    • افتح Resend → Logs لمشاهدة حالة التسليم الفعلية.

صفحة /inquiry/[token] تعرض 404

  1. افتح الـ submission في Studio → اقرأ قيمة "رابط العميل (Access Token)".
  2. قارنها بالـ token في الـ URL — لازم يكونا متطابقين.
  3. لو مختلفين: الـ submission ربما اتسجّلت قبل ميزة accessToken. ابعت رسالة جديدة للاختبار.

الرد لا يظهر في صفحة المحادثة رغم Publish

  1. اعمل refresh للصفحة (Ctrl+Shift+R).
  2. افتح الـ submission في Studio وتأكد إن replyMessage فيه نصّ ومُنشَر (مش draft).

Build يفشل على Vercel

  1. Vercel → آخر deployment → Build Logs.
  2. شغّل npm run build محليًا لاكتشاف نفس الخطأ.
  3. أغلب الأخطاء TypeScript — راجع الـ overloads لـ client.fetch (Sanity v7).

Sanity webhook يـ retry بشكل لا نهائي

  1. عند فشل الـ webhook، Sanity يعيد المحاولة كل ثوانٍ مما يُغرق Vercel logs.
  2. الحل المؤقت: Disable الـ webhook من sanity.io/manage.
  3. إصلاح المشكلة الأصلية (env var أو deploy فاشل).
  4. Reset الـ submission في Studio (امسح replySentAt وreplyError) وPublish.
  5. Enable الـ webhook ثانيةً.

14. الصيانة الدورية

تجديد الـ domain

سنوي — Namecheap يُذكّر بالبريد. التكلفة ~$10/سنة.

تجديد ترخيص Resend

الخطة المجانية تكفي لـ 3,000 إيميل/شهر و100 إيميل/يوم. إذا تجاوزت — Resend يفصل الإرسال حتى نهاية الفترة، أو يمكن الترقية.

سعة Sanity

الخطة المجانية تشمل: 10K documents، 1GB assets. راقب الـ usage من sanity.io/manage → Usage.

سعة Vercel

الخطة Hobby تكفي لمشاريع شخصية. راقب bandwidth من Vercel → Usage.

تحديث الـ dependencies

cd anwar-blog
npm outdated
npm update    # تحديث minor
# للتحديثات الكبرى:
npm install package@latest

أرشفة الرسائل القديمة

في Studio → النماذج المرسلة → غيّر status إلى "أُرشفت" للرسائل المكتملة. يمكن إضافة filter لاحقًا لإخفاء المؤرشفة عن قائمة العرض الافتراضية.
